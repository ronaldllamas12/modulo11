Deployment guide: Render (backend), Vercel (frontend), Supabase (Postgres)

Overview
--------
This guide shows the manual steps to deploy the project components:

- Backend: Render (use the existing `backend/Dockerfile`)
- Frontend: Vercel (Vite static site)
- Database: Supabase (Postgres)

Prerequisites
-------------
- A Supabase project (or Postgres DB) with a connection string.
- Accounts on Render and Vercel.
- `psql` or the Supabase SQL editor to apply migration SQL.

1) Prepare the database (Supabase)
---------------------------------
1. Create a new Supabase project.
2. Get the database connection string (postgresql://user:pass@host:port/dbname).
3. In Supabase SQL editor, run the migration SQL file:

   - Open `backend/migrations/0001_add_area_and_shift_setup.sql` and execute it.

   Or from your machine with `psql`:

```bash
psql "postgresql://<user>:<pass>@<host>:5432/<db>" -f backend/migrations/0001_add_area_and_shift_setup.sql
```

4. Verify tables exist (machines, areas, shifts, shift_machine_setups, events).

2) Deploy backend to Render
---------------------------
1. Create a new Web Service on Render using the Dockerfile option.
   - Repository: Point to this Git repo
   - Root: `backend`
   - Dockerfile: uses `backend/Dockerfile`
2. Set environment variables (in Render service settings):
   - `ENVIRONMENT=production`
   - `DATABASE_URL` = your Supabase connection string (use the `psycopg` scheme for Python 3.14+ or `asyncpg` for older runtimes as configured in `app/core/config.py`)
   - `SECRET_KEY` = strong random value
   - `ALLOWED_ORIGINS` = your frontend URL (comma-separated if multiple)
   - Optional: Cloudinary vars if you use cloud uploads
3. Deploy; Render will build the Docker image and run the service.

Notes:
- The backend Dockerfile uses the `PORT` env var (Render sets it automatically).
- Ensure the database is reachable from Render (Supabase enables external connections by default).

3) Deploy frontend to Vercel
---------------------------
1. Create a new project in Vercel and import this repo, selecting the `frontend` directory as the project root.
2. Vercel will use the `build` script in `frontend/package.json` (`npm run build`).
3. Set environment variables in Vercel if frontend needs them (e.g., `VITE_API_BASE_URL`), for example:
   - `VITE_API_BASE_URL=https://your-backend.onrender.com/api/v1`
4. Deploy. Vercel will publish the static files to a CDN.

4) Post-deploy checks
---------------------
- Call `GET /api/v1/machines` (or open the UI) to confirm the backend connects to the DB and returns data.
- Confirm CORS is configured properly: set `ALLOWED_ORIGINS` to your Vercel domain.

Security notes
--------------
- `SECRET_KEY` is required in production. Do NOT use the default insecure key.
- Back up your database before running migrations.

Troubleshooting
---------------
- If you see missing column errors after deploy, ensure the migration SQL was applied against the same database the backend uses.
- If `SECRET_KEY` errors appear, confirm the env var is set in Render.
